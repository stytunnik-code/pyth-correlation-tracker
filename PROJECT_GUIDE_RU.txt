PYTH CORRELATION TRACKER
ПОЛНОЕ ОПИСАНИЕ ПРОЕКТА, ИЗМЕНЕНИЙ И ПРАВИЛ ИСПОЛЬЗОВАНИЯ

==================================================
1. ЧТО ЭТО ЗА ПРОЕКТ
==================================================

Это веб-сервис для анализа связей между активами на основе данных Pyth.

Проект показывает:
- линейную корреляцию между активами;
- динамику корреляции во времени;
- lead-lag связи, то есть кто двигается раньше, а кто позже;
- энтропию доходностей;
- скрытые нелинейные связи через MI / NMI;
- состояние live feed и базовую диагностику качества данных.

Источники данных в проекте:
- Pyth Hermes: live price updates;
- Pyth Benchmarks: исторические OHLC бары;
- Binance klines: вспомогательный источник для отдельных графиков через proxy endpoint.

Главная идея проекта:
не просто показать цену, а показать, как разные активы связаны между собой, кто за кем двигается, насколько движение хаотичное и где есть скрытая зависимость.


==================================================
2. КАК ЭТО РАБОТАЕТ В ОБЩИХ ЧЕРТАХ
==================================================

2.1. Live данные

Проект регулярно получает live price updates через Pyth Hermes.
Каждая новая live цена теперь сохраняется не только как значение, но и как timestamped tick:

- цена;
- timestamp;
- серия ticks по каждому активу.

Это важно, потому что старая версия часто работала просто с массивами цен без нормального выравнивания по времени.


2.2. Исторические данные

История баров берется из Pyth Benchmarks:
- 1m;
- 1h;
- 4h;
- 1d.

Эти бары используются для:
- ChartView;
- Correlation View;
- Lead-Lag View;
- Entropy View.


2.3. Смешивание истории и live

В новой версии live данные больше не подмешиваются грязно.

Сейчас используется нормальная схема:
- исторические бары берутся из Benchmarks;
- live ticks хранятся отдельно;
- при необходимости они ресемплируются в bar buckets;
- затем merge идет по timestamp buckets, а не по выдуманным временам.

Это устранило большой класс ложных сигналов и визуальных багов.


==================================================
3. КАК ПОЛЬЗОВАТЬСЯ ПРОЕКТОМ
==================================================

3.1. Matrix

Раздел Matrix показывает тепловую карту корреляций между активами.

Что видно:
- пары активов;
- силу связи от -1 до +1;
- топ положительных и топ отрицательных пар;
- среднюю корреляцию по видимому набору активов.

Как использовать:
- выбрать нужные активы через coin picker;
- сузить категорию фильтром;
- отсортировать карточки;
- кликнуть по ячейке heatmap для детального pair analysis.

Что важно понимать:
- здесь используется Pearson correlation;
- теперь корреляция считается по aligned returns, а не по сырым уровням цены;
- это делает результаты статистически чище.


3.2. Charts

Раздел Charts показывает свечной график выбранного актива.

Как использовать:
- выбрать актив;
- выбрать таймфрейм;
- переключать тип отображения;
- смотреть историю и текущую цену.

Что важно:
- график строится на benchmark bars;
- live marker теперь показывается аккуратнее;
- последняя свеча больше не должна рисоваться ложным synthetic баром;
- flat / doji bars теперь не ломают рендер.


3.3. Correlation

Раздел Correlation показывает связь двух активов глубже, чем просто heatmap.

Что есть:
- scatter plot returns;
- rolling correlation;
- выбор пары;
- выбор окна 1D / 7D / 30D / 90D.

Как использовать:
- выбрать актив A;
- выбрать актив B;
- выбрать окно анализа;
- смотреть scatter и rolling correlation.

Что показывает:
- насколько returns движутся вместе;
- меняется ли связь во времени;
- есть ли режимы усиления или ослабления корреляции.


3.4. Lead-Lag

Раздел Lead-Lag показывает, какой актив двигается раньше.

Что есть:
- cross-correlation по разным лагам;
- лучший лаг;
- zero-lag correlation;
- out-of-sample check;
- ranking по парам.

Как использовать:
- выбрать пару активов;
- выбрать окно;
- посмотреть:
  кто лидер;
  кто фолловер;
  на каком лаге связь максимальна;
  подтверждается ли лаг вне train sample.

Что означают поля:
- PEAK CORR: лучшая корреляция на train части;
- OOS CHECK: проверка этого же лага на out-of-sample части;
- ZERO-LAG CORR: корреляция без сдвига;
- PREDICTIVE GAIN: выигрыш лага против нулевого сдвига.

Если OOS CHECK слабый или отвергнут:
- сигнал неустойчивый;
- это не стоит трактовать как надежную predictive связь.


3.5. Gaussian Entropy

Раздел Entropy показывает, насколько движение returns структурное или хаотичное.

Что есть:
- ranking активов по gaussian entropy;
- adjusted NMI heatmap;
- hidden connections;
- live measuring режим;
- correlation stability alert.

Как использовать:
- выбрать активы;
- запустить live measuring;
- смотреть:
  какие активы более предсказуемые;
  какие более хаотичные;
  где есть нелинейные зависимости.

Что означает ranking:
- низкая gaussian entropy = returns более сжаты и регулярны;
- высокая gaussian entropy = returns более шумные и хаотичные.

Что означает adjusted NMI:
- показывает нелинейную зависимость между активами;
- baseline shuffled noise вычтен;
- поэтому сигнал стал чище, чем в старой версии.

Что означают hidden connections:
- пары, где обычный Pearson может показывать слабую связь;
- но adjusted NMI показывает, что структура зависимости все равно есть.


3.6. Feed Diagnostics

В header теперь есть badge состояния feed:
- FEED OK;
- FEED WARN;
- FEED DEGRADED.

По клику открывается diagnostics popover.

Что там видно:
- проблемные symbols;
- age;
- skew;
- порядок timestamps;
- тип проблемы.

Это нужно для быстрой проверки, не деградировал ли live feed.


==================================================
4. ЧТО ИМЕННО ИЗМЕНИЛОСЬ ПО СРАВНЕНИЮ С V1
==================================================

Ниже перечислены основные изменения относительно старой версии.


4.1. Корреляция в Matrix

V1:
- корреляция считалась по сырым уровням цены;
- live series не были нормально выровнены по времени;
- это давало ложные корреляции.

Сейчас:
- correlation считается по aligned returns;
- используется timestamped live store;
- шум и ложные связи стали меньше.

Итог:
Matrix стала статистически намного честнее.


4.2. Работа с live ticks

V1:
- timestamps местами были фейковыми или косвенными;
- series могли склеиваться по индексу, а не по времени.

Сейчас:
- live ticks сохраняются как timestamped points;
- merge history + live идет по реальным bucket timestamps;
- есть sanity layer для stale/skew/order.

Итог:
live pipeline стал намного чище.


4.3. Correlation View

V1:
- live хвост мог домешиваться грязно;
- были риски ложной rolling dynamics.

Сейчас:
- 1D окно использует корректный merge benchmark bars + live ticks;
- fake timestamp tail убран.

Итог:
rolling correlation стал более правдоподобным.


4.4. Lead-Lag

V1:
- лучший лаг выбирался и оценивался на одном и том же окне;
- это был in-sample bias;
- ranking переоценивал шумовые сигналы.

Сейчас:
- лаг сначала ищется на train части;
- потом проверяется на out-of-sample;
- ranking теперь опирается на validatedCorr, а не только на peak in-sample.

Итог:
lead-lag сигналы стали менее фальшивыми.


4.5. Entropy / NMI

V1:
- в UI и логике была путаница между Shannon entropy и Gaussian entropy;
- NMI был слишком шумным;
- hidden connections могли всплывать даже на почти случайных данных.

Сейчас:
- ranking явно трактуется как Gaussian entropy of returns;
- добавлен permutation-adjusted NMI;
- скрытые связи теперь чище и менее шумные.

Итог:
Entropy раздел стал математически более последовательным.


4.6. Pearson

V1:
- при нулевой дисперсии Pearson мог вернуть fake 0;
- это маскировало degenerate cases.

Сейчас:
- non-finite значения фильтруются;
- degenerate series дают null.

Итог:
меньше ложной уверенности в корреляции.


4.7. Returns

V1:
- были риски деления на 0;
- non-finite значения могли протекать дальше в аналитику.

Сейчас:
- pctReturns защищен от 0 и bad values.

Итог:
цепочка calculations стала устойчивее.


4.8. HYPE chart bug

V1:
- последняя свеча HYPE могла выглядеть сломанной;
- были ложные гэпы;
- marker и last candle могли визуально конфликтовать.

Сейчас:
- synthetic live candle logic поджата;
- zero-range bars рендерятся корректно;
- marker показывается осторожнее;
- старые сценарии с ложным разрывом сильно снижены.


4.9. API proxy

V1:
- `/api/pyth` мог падать whole-response, если падал один batch.

Сейчас:
- используется `Promise.allSettled`;
- partial success больше не валит весь response.


4.10. Sanity diagnostics

V1:
- не было быстрого способа понять, что feed уже сомнительный.

Сейчас:
- есть live-data health utilities;
- есть tests;
- есть runtime badge;
- есть diagnostics popover.

==================================================
6. КАК ПРАВИЛЬНО ИНТЕРПРЕТИРОВАТЬ РЕЗУЛЬТАТЫ
==================================================

6.1. Correlation

Высокая корреляция не означает причинность.

Это значит только то, что returns двух активов часто двигаются похоже.


6.2. Lead-Lag

Даже если есть красивый лаг, это не гарантирует торговое преимущество.

Смотреть надо на:
- OOS CHECK;
- стабильность сигнала;
- размер выборки;
- режим рынка.


6.3. Entropy

Низкая entropy не означает, что актив "хороший".

Это означает только то, что его returns в выбранном окне более структурны и менее шумны.


6.4. Adjusted NMI

Высокий adjusted NMI не означает, что активы линейно повторяют друг друга.

Это означает:
между ними есть зависимость, которую Pearson может не ловить.


==================================================
7. ОГРАНИЧЕНИЯ ТЕКУЩЕЙ ВЕРСИИ
==================================================

Несмотря на большой объем улучшений, ограничения остаются.

Что еще не идеально:
- нет полноценной backend telemetry системы;
- нет e2e browser tests;
- часть UI текстов может требовать косметической правки;
- divergence между live и benchmark пока не вынесен в отдельную полноценную политику отображения;
- Lead-Lag validation базовая, а не академически полная;
- MI estimator все еще упрощенный, хоть и очищенный baseline correction.


==================================================
8. КОГДА СЧИТАТЬ, ЧТО FEED ПЛОХОЙ
==================================================

На практике feed стоит считать проблемным, если:
- badge показывает FEED WARN или FEED DEGRADED;
- у нескольких активов stale ticks;
- много timestamp skew;
- есть non-monotonic ticks;
- графики выглядят рваными или цена явно не обновляется.

Главные улучшения:
